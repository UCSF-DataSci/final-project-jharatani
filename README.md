# DATASCI_223_Final_Project_W26

## Overview of the Problem
Effectively asking if a general LLM can interpret text produced/generated by another LLM, especially text built specifically to provide more non-standard terminology, abbreviations, complex phrases, and frequent misspellings in order to develop more robust systems for disease entity identification and linking in biomedical and clinical text. This project on a broader scale allows us to see model-to-model transfer of information; with complex terminologies a lot can get lost in translation. Overall, this project is a continuation of the project the original dataset was produced for, which was possibility for enhancement of the development of more robust systems for disease entity identification. 

## Dataset
Using dataset from **Synthetic Mention Corpora for Disease Entity Recognition and Normalization Study**, found via PhysioNet (thx Chris). This dataset and the study associated aimed to address current limitations in training data availability in the biomedical domain.
**Please note the dataset was very large - so to view entire dataset please refer to DOI link on physionet - it is provided publicly**

The dataset is a comma separated (csv) file with two columns: "cui" and "matched_output".

"cui" column contains the UMLS CUI for disease highlighted in the synthetic mention

The "matched_output" column contains the synthetic mention with the disease name highlighted between two tags <1CUI> and </1CUI>.

## Methods/Approach
We use a pretrained large language model to perform structured information extraction on a synthetic biomedical corpus originally generated by a fine-tuned LLaMA-2 model. No additional model training or fine-tuning was performed; all results are obtained via prompt-based inference. After the information extraction, we moved to build an agent that could use this information to respond to queries and produce a triage/pseudo-diagnosis/course of action at a pre-medical professional level. 

**Pt. 1**

The model takes the synthetic provided data in csv format and outputs standardized JSON fields (diagnosis, medications, lab values, confidence), producing a machine-readable dataset for downstream semantic search. 

**Pt. 2**

This is where we have merged our json file output diagnoses and performed a semantics search to further assess the data. More specifically, we have loaded the extracted records from Pt.1, converted them into embedding vectors using a local model, and built a searchable corpus. If given a query, the local model was able to find the most semantically similar notes using cosine similarity.

**Pt. 3**

Now we add an agent to the mix, and ask it natural language questions about the output from Pt.1/2. The expectation is for the agent to look at the query, match any records to the query based on key words or indications and provide a triage level, red flags, next steps and a disclaimer. So if we provide a query mentioning a patient with heart attack symptoms and high WBC, it will find records describing similar circumstances, further assess the clinical notes, and come up with a course of action.

## Design Decisions & Trade-offs
Some issues with the dataset from the get-go as mentioned in the original paper is the presence of potential hallucinations, lower quality than real clinical notes, and possible lack of realistic content around the disease span. With no PHI in the data, we still decided to move forward and provide guardrails to ensure no potential PHI came up in the responses produced by our agent. We additionally added guardrails for hallucinations, but since there were already hallucinations in the dataset to begin with, that was a challenge in and of itself. The purpose in providing the guardrails anyways is to be able to use this general framework in the future, maybe when assessing different datasets that have real clinical data.

We added the original csv to .gitignore due to size and github limitation on how much you can push in one go.
We decided to use a local model in our Pt.1 steps in training the LLM and outputting a structured format in order to avoid using Open Router AI.
We limited the number of notes (small sample size) to extract and structure in the JSONL due to runtime constraints and testing the model initially. This may have caused the agent to miss relevant information simply because not enough records were sampled.
Adding more examples/few shot prompting helped develop more concise and accurate outputs, additionally helped understand scenarios with fragmented matched_outputs. This may have caused bias though.

## Example Output

Query: patient with heart attack symptoms and high WBC
Response:
Based on the semantic search results, I found two potential notes that mention heart attack symptoms and high WBC levels.

The first note is related to a patient with coronary artery disease, hyperlipidemia, hypertension, and osteoarthritis. The second note mentions pulmonic valve replacement, left atrial fibrillation, myotonia, potassium-aggravated, peripheral arterial disease, aortic stenosis, chronic obstructive pulmonary disease, and piperacillin-tazobactam and levofloxacin for a CTCAE grade 4 arthritis.

However, the most relevant note to both symptoms is:

**C1560000**

This note mentions chest x-ray showing a right upper lobe opacity, which could be indicative of a heart attack. It also mentions that the patient was started on piperacillin-tazobactam and levofloxacin for a CTCAE grade 4 arthritis.

The triage level is **URGENT** due to the presence of high WBC levels, which may indicate an infection or inflammation.

Red flags:

* High WBC levels
* Chest x-ray showing a right upper lobe opacity

Next steps:
1. Further evaluation and testing for infections or inflammatory conditions.
2. Consider administering antibiotics or other treatments as needed.
3. Monitor the patient's condition closely and adjust treatment as necessary.
